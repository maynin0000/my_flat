# Extractor Spec v0

Extractor Spec v0 (WebView 관측 기반)
0. 목적

RN(WebView)에서 무신사 상품 상세 화면을 “사용자 디바이스에서 열고”

가격(정가/판매가/할인율) + 상품 상태 + 사이즈별 품절 여부를 추출하여

서버로 Snapshot 업로드

서버는 스냅샷 차이를 비교해 재입고 이벤트(sold_out→in_stock) 및 가격 이벤트를 생성

1. 스냅샷에서 “진짜” 필요한 필드 정의

MVP 핵심 기능(재입고 알림 + 가격/쿠폰 예상가)을 만족시키려면 최소 스키마는 다음이야.

1.1 Snapshot (공통 메타)

platform: "musinsa"

source_url: 원본 URL

product_key: 플랫폼 내 상품을 식별할 키(가능하면 goods_no 등), 없으면 URL 기반 fallback

captured_at: ISO timestamp

extractor_version: "musinsa_v0_dom" 같은 버전 문자열

confidence: 0~1 (추출 신뢰도)

last_checked_at: (=captured_at)

1.2 Price

sale_price: number (필수)

original_price: number | null

discount_rate: number | null (없으면 계산 가능)

currency: "KRW" (기본값)

1.3 Stock

product_status: "on_sale" | "sold_out" | "ended" | "unknown"

sizes: { [sizeLabel: string]: "in_stock" | "sold_out" | "unknown" }

핵심: unknown을 1급 상태로 인정
B안(관측 기반) UX랑 잘 맞고, 데이터가 안 잡히는 케이스도 “정직하게” 표현 가능.

2. 추출 전략 개요 (2단 + 폴백)

MVP는 “일단 돈는 것”과 “유지보수 가능한 것”을 동시에 잡아야 해서 전략을 이렇게 둔다.

2.1 1차: DOM 기반 추출 (musinsa_v0_dom)

WebView가 렌더링 완료된 DOM에서

가격 텍스트

품절/판매중지 배지

사이즈 옵션 UI 상태

를 읽어서 스냅샷 생성

장점: 구현 빠름, 디버깅 쉬움
단점: UI/클래스 변경에 취약

2.2 2차(확장): 네트워크 응답 훅 기반 추출 (musinsa_v1_net)

WebView 내부에서 fetch / XHR를 래핑하여

옵션/재고 관련 JSON 응답을 탐지해 파싱

장점: 재고 정확도/안정성↑
단점: 구현 난이도↑ (그래서 v1)

2.3 폴백(B 방식): 사용자 보정/확인 요청

추출 실패(unknown 많음) 또는 마지막 확인 오래됨 → “지금 확인하기” 유도

완전 안 되면 사용자에게 “사이즈 상태 수동 체크” 입력 옵션 제공(최후의 보조)

3. DOM 추출 규칙 (v0 핵심)
3.1 로드/타이밍 규칙

DOM은 타이밍이 생명이라, 아래 규칙을 강제한다.

페이지 로딩 후 “안정화 대기”:

document.readyState === "complete"만으로 부족할 수 있음

추가로 setTimeout 또는 “핵심 요소 존재” 조건을 만족할 때까지 대기

최대 대기: 예) 8초

8초 동안 핵심 요소 못 찾으면 confidence 낮추고 unknown 처리

추출 트리거

1회성: “지금 확인하기” 버튼

자동: 앱 진입 시 “관측 큐”에서 1~N개 순차 갱신

3.2 가격 추출 규칙
3.2.1 기본 원칙

화면에 표시된 가격은 “문자열”이므로 정규화가 필요

금액은 다음을 처리해야 함:

천 단위 콤마: 89,000

원/₩/공백: 89,000원, ₩89,000

범위 표현/옵션별 가격(예외): 89,000 ~ 99,000

3.2.2 파싱 함수 규격

입력: string

출력: number | null

규칙:

숫자와 콤마만 남기고 제거

콤마 제거 후 parseInt

값이 0 또는 비정상적으로 큰 값(예: 10억 이상)이면 null 처리(가드레일)

3.2.3 sale_price, original_price 선택 규칙

우선순위:

“현재가/판매가”로 의미가 명확한 영역

그 다음 “큰 글씨로 표시된 가격”

original_price:

“정가/할인 전 가격”이 별도로 보이면 추출

없으면 null

discount_rate:

화면에 % 표기가 있으면 추출

없으면 original_price와 sale_price로 계산:

round((1 - sale/original) * 100)

original이 null이면 null

따질 포인트: “쿠폰가” 같은 개인화 가격이 화면에 보이는 경우
→ 우리 MVP는 개인 쿠폰을 “유저 입력 기반 예상가”로 제공하므로, DOM에서 쿠폰가가 보여도 sale_price와 혼동하지 않게 label 우선으로 읽어야 한다.

3.3 상품 상태(product_status) 추출 규칙

가장 위험한 오판 포인트라 규칙을 명확히 둔다.

"ended": 판매종료/판매중지/품절과 다른 “더 이상 판매 안 함” 신호가 있을 때

"sold_out": 상품 전체 품절 또는 모든 옵션 품절로 추정될 때

"on_sale": 판매 가능 신호가 명확할 때

"unknown": 애매하면 무조건 unknown

판정 우선순위

페이지에 “판매종료/판매중지” 명시 → ended

전체 “품절” 명시 → sold_out

구매/장바구니 버튼이 활성화 + 품절 배지 없음 → on_sale

그 외 unknown

따질 포인트: “일부 사이즈 품절”인데 상품 배지가 품절로 뜨는지?
→ 이 경우 product_status는 on_sale이면서 size별 sold_out이 섞일 수 있다.
“전체 품절”은 구매 불가인지로 판정.

3.4 사이즈/옵션 추출 규칙 (v0의 가장 중요한 부분)
3.4.1 sizeLabel 정규화

사이즈는 텍스트가 다양함: S, M, L, 90, 95, FREE, One Size, OS 등

sizeLabel은 “화면에 표시된 라벨” 그대로 저장(원문 유지)

다만 정렬/비교를 위해 sizeKey(정규화 키)를 추가로 만들 수도 있음(확장)

3.4.2 옵션 리스트 찾기(탐색 전략)

DOM에서 “옵션/사이즈”는 보통 버튼 리스트, 셀렉트 박스, 바텀시트 형태로 존재.
그래서 탐색은 다음처럼 “형태 기반”으로 한다.

(1) 버튼/리스트 형태: 클릭 가능한 요소가 반복되고 텍스트가 사이즈 후보

(2) select/option 형태: <select> 내부 옵션 텍스트가 사이즈 후보

(3) 모달/바텀시트 형태: “사이즈 선택” 버튼을 눌러야 리스트 생성

v0에서는 (1)(2)를 먼저 처리하고, (3)은 “지금 확인하기” 누르면 자동으로 사이즈 선택 UI를 한번 열었다가 닫는 방식까지 고려 가능(단, UX 영향/부작용 검토 필요)

3.4.3 sold_out 판정 규칙(형태 기반)

옵션이 품절인지 판단할 때, 셀렉터에 의존하지 말고 “상태 신호”를 여러 개 본다.

우선순위 신호(하나라도 만족하면 sold_out 쪽 가중치):

disabled 속성

aria-disabled="true"

클래스/데이터 속성에 soldout|sold-out|outofstock|disabled 등 키워드 포함

텍스트에 “품절” 포함

스타일(회색 처리/취소선)도 신호가 될 수 있지만 v0에서는 보조로만

판정 방식

각 옵션 요소마다 soldOutScore를 계산:

disabled/aria-disabled: +3

class 키워드: +2

텍스트 “품절”: +3

스타일 신호: +1

score >= 3이면 sold_out, 0이면 in_stock, 애매하면 unknown

이 스코어링은 “UI 변경 내성”을 높이는 장치야.
특정 클래스 하나만 믿으면 바로 깨짐.

3.4.4 “옵션이 안 보이는” 케이스 처리

사이즈가 없는 상품(FREE only)

sizes = { "FREE": in_stock } 형태로 저장 가능

옵션 렌더링 실패/로그인 요구/성인인증/품절 알림 페이지 등

sizes는 빈 객체 {}가 아니라, unknown을 명시적으로 담거나

최소 product_status만이라도 뽑아 confidence 낮게 업로드

4. 신뢰도(confidence) 계산 규칙

B안 UX에서 “마지막 확인 시간”과 함께, 내부적으로는 신뢰도가 중요해.

예시 규칙(0~1):

sale_price 존재: +0.4

original_price 또는 discount_rate 존재: +0.1

sizes에서 unknown 비율이 낮음: +0.4 (unknown 0%면 만점, 50%면 절반 등)

product_status가 unknown이 아님: +0.1

최대 1.0

활용

confidence < 0.6이면:

UI에 “확인이 필요할 수 있어요” 표시

(서버) 재입고 판단은 하되, 알림 메시지에 “최신 확인 시각” 강조

5. 실패/예외 케이스 목록 (반드시 대비)

여기서부터 “따져야 할 것들”의 핵심.

5.1 URL 형태 예외

상품 상세가 아닌 링크(카테고리/검색/이벤트)

리다이렉트 링크

앱 딥링크 스킴(musinsa://)

글로벌/모바일/PC 도메인 혼재

정책

등록 단계에서 URL 정규화:

가능한 한 “상품 상세 canonical URL”로 통일

실패 시 사용자에게 “상품 상세 링크를 넣어주세요” 안내

5.2 접근 제한

로그인 필요 페이지

성인 인증

지역/앱 전용

비정상 트래픽 차단 페이지(캡차 등)

정책

감지되면 product_status unknown + reason 코드 저장:

blocked_login_required, blocked_captcha, blocked_age_gate 등

유저에게는 “확인할 수 없는 페이지”로 표시

5.3 렌더링 지연/스크립트 로딩 실패

readyState complete인데 실제 데이터는 로딩 중

네트워크 느림

정책

“핵심 요소 발견” 기반 wait

최대 대기 후 추출 시도 → confidence 낮추고 업로드

6. 데이터 업로드 정책 (앱→서버)
6.1 업로드 트리거

사용자가 “지금 확인하기”

알림 탭으로 들어왔을 때 자동 1회

앱 실행 시: 트래킹 리스트 중 “우선순위 상위 N개” 갱신

6.2 업로드 빈도 제한(디바이스)

같은 상품은 기본 1~3시간 쿨다운(디폴트는 나중에 조정)

유저가 수동으로 “지금 확인” 눌렀을 때는 쿨다운 무시 가능(단, 서버 rate limit)

6.3 서버 중복 제거(필수)

동일 captured_at 근처(예: 5분 이내) 동일 데이터면 저장 생략 가능

단, “상태 변화 이벤트” 감지 목적이므로 타임스탬프/순서를 유지할 최소 저장 정책 필요

7. 관측 기반 재입고 판단을 위한 서버 요구사항(Extractor 관점)

Extractor는 서버의 디프 로직이 전제되어야 의미가 생긴다.

서버는 상품+사이즈별 이전 상태를 기억

새 스냅샷에서 sold_out→in_stock이면 이벤트 생성

이벤트는 디듀프/쿨다운 적용(우리가 디폴트로 1시간)

8. 디버깅/운영을 위한 “로그/리플레이” 설계(강추)

유지보수의 80%는 여기서 결정난다.

8.1 앱에서 서버로 보내는 debug_payload (옵션)

추출에 사용한 “근거 신호”를 일부 전송:

발견한 가격 텍스트 원문

옵션 개수

unknown 비율

reason 코드(차단/로그인/캡차 등)

개인정보/세션 정보는 절대 포함 금지

8.2 리플레이를 위한 HTML 스냅샷? (비권장/주의)

페이지 전체 HTML을 서버로 보내면 법/보안/용량 리스크 큼

대신 “핵심 텍스트/구조 요약”만 전송하는 방식 추천

9. 테스트 플랜 (MVP 전에 꼭)
9.1 케이스 세트

일반 상품(사이즈 있음)

FREE 상품

일부 사이즈 품절 상품

전체 품절 상품

판매종료/판매중지 상품

성인/로그인 제한 페이지(있다면)

가격 변동 케이스(세일 중)

9.2 품질 기준

sale_price 추출 성공률: 95%+

사이즈 목록 추출 성공률: 85%+ (v0 기준, v1에서 올림)

사이즈 품절 판정 정확도: 85%+ (v0), 95% 목표(v1)
